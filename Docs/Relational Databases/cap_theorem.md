# What Is the CAP Theorem? (ทฤษฎี CAP คืออะไร?)

> ⚖️ **แหล่งที่มา:** [IBM — What Is the CAP Theorem?](https://www.ibm.com/think/topics/cap-theorem)

## 1. CAP Theorem คืออะไร?
**CAP Theorem (หรือ Brewer’s Theorem)** เป็นกฎเหล็กของการออกแบบระบบแบบกระจายศูนย์ (Distributed Systems) ที่กล่าวไว้ว่า:
> **"ระบบ Distributed System สามารถเลือกคุณสมบัติที่ดีที่สุดได้เพียง 2 ใน 3 อย่างนี้เท่านั้น (Consistency, Availability, หรือ Partition tolerance)"**

เปรียบเทียบง่ายๆ เหมือนเวลารับเหมาก่อสร้างที่มักจะมีกฎว่า "ถูก, เร็ว, หรือ ดี — คุณเลือกได้แค่ 2 อย่าง" (เช่น ถ้าจะเอาถูกและเร็ว งานก็จะไม่ออกมาดี)

---

## 2. ตัวอักษร C, A, และ P หมายถึงอะไร?

### 🪞 C: Consistency (ความสม่ำเสมอของข้อมูล)
หมายความว่า **"Client ทุกคนต้องเห็นข้อมูลล่าสุดที่ตรงกันเสมอ ไม่ว่าจะดึงข้อมูลมาจาก Node ไหนก็ตาม"** 
*   **เงื่อนไข:** เมื่อมีการเขียนข้อมูลลง Node หนึ่ง ข้อมูลนั้นต้องถูกคัดลอก (Replicate) ไปยัง Node อื่นๆ ในระบบทั้งหมดให้เสร็จสิ้นก่อน จึงจะนับว่าการเขียน "สำเร็จ"

### 🟢 A: Availability (ความพร้อมใช้งาน)
หมายความว่า **"ทุกคำขอ (Request) ต้องได้รับการตอบกลับเสมอ แม้ว่าจะมีบาง Node ในระบบพังอยู่ก็ตาม"**
*   **เงื่อนไข:** Node ที่ยังรอดชีวิตอยู่ ต้องตอบกลับข้อมูลได้ทุกครั้งแบบไม่มีข้อยกเว้น ไม่มีการปล่อยให้ระบบค้างหรือขึ้น Error ว่าไม่พร้อมใช้งาน

### ✂️ P: Partition tolerance (ความทนทานต่อการตัดขาดของเครือข่าย)
"Partition" คือสถานการณ์ที่เครือข่ายระหว่าง 2 Nodes ถูกตัดขาดหรือคุยกันไม่ได้ชั่วคราว (Network break) 
*   **เงื่อนไข:** ระบบโดยรวมต้อง **ยังคงทำงานต่อไปได้** แม้ว่าจะเกิดการตัดขาดการเชื่อมต่อระหว่าง Node จำนวนเท่าใดก็ตาม

## 3. รูปแบบฐานข้อมูลตามทฤษฎี CAP

ในโลกของระบบกระจายศูนย์ (Distributed Systems) อาการเครือข่ายหลุด (Partition) เป็นสิ่งที่ "หลีกเลี่ยงไม่ได้" ดังนั้นเราจึง**ต้องรวม P ไว้เสมอ** หมายความว่าในการเลือกฐานข้อมูล NoSQL เราจึงเลือกได้แค่ **CP** หรือ **AP** เท่านั้น:

1.  **CP (Consistency + Partition tolerance):**
    *   *เน้นความถูกต้อง:* ยอมทิ้งความพร้อมใช้งาน (Availability) ไป หากระบบเจอ Network Partition (Node คุยกันไม่ได้) Node ที่มีปัญหาจะถูกสั่งให้ "หยุดให้บริการชั่วคราว" จนกว่าเครือข่ายจะกลับมา เพื่อป้องกันปัญหาข้อมูลไม่ตรงกัน
    *   *ตัวอย่าง:* **MongoDB** (เป็นระบบมี Master เดียว ถ้า Master ล่ม จะหยุดรับข้อมูลชั่วคราวจนกว่าจะเลือก Master ใหม่เสร็จ)
2.  **AP (Availability + Partition tolerance):**
    *   *เน้นพร้อมใช้งาน:* ยอมทิ้งความสม่ำเสมอ (Consistency) ไป หากเจอ Network Partition ทุก Node จะยังคงรับแขกและคืนข้อมูลให้ Client เสมอ อาจะทำให้บางคนได้ข้อมูลเก่าไปบ้าง (Stale data) แต่พอเน็ตเวิร์กกลับมาปกติ ระบบจะทำการซิงค์ข้อมูลให้ตรงกันในภายหลัง (เรียกว่า **Eventual Consistency**)
    *   *ตัวอย่าง:* **Apache Cassandra**, CouchDB
3.  **CA (Consistency + Availability):**
    *   *ตัวละครลับ:* ถ้าจะเอาทั้งตรงกันและพร้อมเสมอ จะไม่สามารถทนทานต่อเครือข่ายขาดได้เลย ซึ่งในโลกความเป็นจริงเป็นไปไม่ได้ใน Distributed System อย่างไรก็ตาม ฐานข้อมูลเชิงสัมพันธ์แบบเดี่ยวๆ เช่น **PostgreSQL** มักถูกจัดในกลุ่มนี้ 

---

## 4. สรุป: จะเลือกใช้อะไรตอนออกแบบ Microservices?

การเข้าใจ CAP Theorem ทำให้คุณเลือกเครื่องมือได้ตรงกับโดเมนของธุรกิจ:
*   ถ้าทำแอประบบตะกร้าสินค้า โซเชียลมีเดีย หรือฟีดข่าว ที่ต้องการให้แอปไม่ล่มและโหลดเร็ว (รับได้ถ้าคอมเมนต์เพื่อนอัปเดตช้าไป 2 วิ) 👉 **ควรเลือก AP (เช่น Cassandra)**
*   ถ้าทำแอประบบธนาคาร ตัดเงิน ยอดเงินในบัญชี (ที่การเห็นข้อมูลผิดไป 10 วินาทีคือหายนะ) 👉 **ต้องเน้น Consistency แบบ CP หรือใช้ Relational DB อย่าง PostgreSQL แทน**
