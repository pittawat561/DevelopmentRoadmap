# ส่วนที่ 1: Database Schema คืออะไร?

> 🗂️ **แหล่งที่มา:** [IBM — What Is a Database Schema?](https://www.ibm.com/think/topics/database-schema)

## 1.1 Database Schema คืออะไร?
**Database Schema (สคีมาฐานข้อมูล)** คือ "แบบแปลนพิมพ์เขียว (Blueprint)" ที่ใช้กำหนดว่าข้อมูลจะถูกจัดระเบียบและเก็บในฐานข้อมูลเชิงสัมพันธ์ (Relational Database) อย่างไร โดยจะครอบคลุมถึง:
*   ชื่อของตาราง (Table names)
*   ชื่อฟิลด์หรือคอลัมน์ (Fields)
*   ชนิดของข้อมูลในแต่ละประโยค (Data types)
*   ความสัมพันธ์ระหว่างตาราง (Relationships)
*   เงื่อนไขข้อจำกัดต่างๆ (Logical constraints)

การออกแบบ Schema นี้กระบวนการมักถูกเรียกว่าการทำ **Data Modeling** ซึ่งบางครั้งจะถูกวาดเป็นแผนภาพ (Diagram) เพื่อให้ทุกฝ่าย (นักพัฒนา, แอดมิน, นักวิเคราะห์) เข้าใจโครงสร้างตรงกัน

---

## 1.2 Schema แตกต่างกับ Database Instance อย่างไร?
*   **Database Schema:** คือ "โครงร่างแบบแปลน" ที่กำหนดว่าข้อมูลเชื่อมโยงกันอย่างไร **แต่ไม่มีข้อมูลอยู่จริงข้างใน** โครงร่างนี้แทบจะไม่ค่อยถูกเปลี่ยนแปลง
*   **Database Instance:** คือ "ข้อมูลฉบับจริง" ที่ถูกบันทึกอยู่ในฐานข้อมูล **ณ วินาทีใดวินาทีหนึ่ง (Snapshot)** ซึ่งจะถูกใส่ ลบ อัปเดต และเปลี่ยนแปลงอยู่ตลอดเวลา

---

## 1.3 ประเภทของ Database Schemas (3 ระดับ)
การออกแบบ Schema มักแบ่งออกเป็น 3 ระดับความลึก:
1.  **Conceptual Database Schema (ระดับคอนเซปต์):** แบบร่างภาพรวมใหญ่ที่สุด ว่าระบบควรมีอะไรบ้างเกี่ยวข้องกันอย่างไร มักอิงตาม Business Rules ทำขึ้นในช่วงแรกของการคุย Requirement งาน
2.  **Logical Database Schema (ระดับตรรกะ):** ลงลึกชึ้น โดยเริ่มกำหนดชื่อตาราง รหัส ฟิลด์ ความสัมพันธ์ และกฎความสมบูรณ์ของข้อมูล (Integrity Constraints) แต่ยัวไม่มีการระบุเครื่องมือหรือเทคโนโลยีที่จะใช้จริง
3.  **Physical Database Schema (ระดับกายภาพ):** นำระดับ Logical มาแปลงเป็นคำสั่งเชิงเทคนิค (Technical information & Syntax) ที่เจาะจงกับ Database ยี่ห้อนั้นๆ (เช่น คำสั่ง SQL สำหรับสร้างตารางจริงบนดิสก์)

---

## 1.4 โครงสร้างแบบ Star Schema vs Snowflake Schema
ในฝั่งของการวิเคราะห์ข้อมูล (เช่น Data Warehouse) มักจะได้ยินการจัดดครงสร้าง 2 แบบหลักๆ ซึ่งใช้ Primary Key และ Foreign Key ในการเชื่อม:

*   ⭐ **Star Schema (ทรงดาว):** มีตารางหลัก (Fact table) อยู่ตรงกลาง และเชื่อมต่อชั้นเดียวออกไปยังตารางมิติ (Dimension tables) รอบๆ
    * *ข้อดี:* เรียบง่ายมากและคิวรีข้อมูลดึงมาดูข้ามตารางรวดเร็ว
*   ❄️ **Snowflake Schema (ทรงเกล็ดหิมะ):** มีตารางหลักตรงกลาง แต่ตารางมิติรอบๆ สามาถแตกแยกย่อยเป็นตารางมิติอื่นๆ ต่อทอดไปได้อีก (Many-to-one relationship)
    * *ข้อดี:* ลดความซ้ำซ้อนของข้อมูล (Redundancy) แลกมากับการ Query ที่จะทำงานได้ช้าลงเพราะต้อง Join ตารางเยอะ

---

## 1.5 ประโยชน์ของการออกแบบ Schema ที่ดี
1.  **Access and Security (ความปลอดภัยและการจัดการสิทธิ):** การแยกตารางชัดเจนทำให้กำหนดสิทธิการเข้าถึงของ User แต่ละคนได้ง่าย เช่น ตารางไหนข้อมูลส่วนบุคคล (PII) สามารถล็อคและเข้ารหัสเฉพาะส่วนได้
2.  **Organization and Communication (การจัดการและการสื่อสาร):** ทำให้ Programmer และ Data Analyst มี Source of truth ว่าตารางไหนเชื่อมกับอะไร ทำงานร่วมกันได้ไม่ผิดพลาด
3.  **Integrity (ความสมบูรณ์ของข้อมูล):** ป้องกันข้อมูลซ้ำซ้อนกัน (Data duplication) ลบปัญหาข้อมูลพัง และทำให้ระบบรักษาคุณสมบัติความเป็นกรด (ACID Properties) ไว้ได้อย่างสมบูรณ์

## 1.6 เนื้อหาเพิ่มเติม: นิยามและรายละเอียดของ Database Schema

### นิยามของ Database schema
Database schema (สกีมาของฐานข้อมูล) คือการนิยามวิธีจัดระเบียบข้อมูลภายในฐานข้อมูลเชิงสัมพันธ์ สิ่งนี้ครอบคลุมถึงข้อจำกัดเชิงตรรกะ เช่น ชื่อตาราง, ฟิลด์, ชนิดข้อมูล และความสัมพันธ์ระหว่างเอนทิตีเหล่านี้

สกีมามักจะใช้การนำเสนอด้วยภาพเพื่อสื่อสารสถาปัตยกรรมของฐานข้อมูล กลายเป็นรากฐานสำหรับการกำกับดูแลการจัดการข้อมูลขององค์กร กระบวนการออกแบบสกีมาฐานข้อมูลนี้เรียกอีกอย่างว่าการสร้างแบบจำลองข้อมูล (Data modeling)

โมเดลข้อมูลเหล่านี้ทำหน้าที่ในหลากหลายบทบาท เช่น ผู้ใช้ฐานข้อมูล ผู้ดูแลระบบฐานข้อมูล และโปรแกรมเมอร์ ตัวอย่างเช่น สามารถช่วยผู้ดูแลระบบฐานข้อมูลจัดการกระบวนการ Normalization เพื่อหลีกเลี่ยงความซ้ำซ้อนของข้อมูล หรืออีกนัยหนึ่ง สามารถช่วยให้นักวิเคราะห์นำทางโครงสร้างข้อมูลเหล่านี้เพื่อจัดทำรายงานหรือการวิเคราะห์ธุรกิจที่มีค่าอื่นๆ ไดอะแกรมเหล่านี้ทำหน้าที่เป็นเอกสารที่มีค่าภายในระบบจัดการฐานข้อมูล (DBMS) เพื่อให้แน่ใจว่ามีความสอดคล้องกันระหว่างผู้มีส่วนได้ส่วนเสียต่างๆ

### สกีมาของฐานข้อมูล เทียบกับ อินสแตนซ์ของฐานข้อมูล
สกีมาของฐานข้อมูลถือเป็น "พิมพ์เขียว" (Blueprint) ของฐานข้อมูลซึ่งอธิบายถึงวิธีที่ข้อมูลอาจเกี่ยวข้องกับตารางอื่นหรือโมเดลข้อมูลอื่น อย่างไรก็ตาม สกีมาไม่ได้บรรจุข้อมูลจริงๆ

ตัวอย่างข้อมูลจากฐานข้อมูล ณ ช่วงเวลาเดียวเรียกว่า อินสแตนซ์ของฐานข้อมูล (Database instance) มันบรรจุคุณสมบัติทั้งหมดที่สกีมาอธิบายไว้เป็นค่าข้อมูล เนื่องจากอินสแตนซ์ของฐานข้อมูลเป็นเพียงภาพรวม (Snapshot) ณ เวลาที่กำหนด จึงมีแนวโน้มที่จะเปลี่ยนแปลงไปตามกาลเวลา ซึ่งแตกต่างจากสกีมาของฐานข้อมูล

### ประเภทของสกีมาฐานข้อมูล (เพิ่มเติม)
* **Conceptual schemas (สกีมาตามระดับแนวคิด):** นำเสนอมุมมองภาพใหญ่ว่าระบบจะมีอะไรบ้าง ข้อมูลจะถูกจัดระเบียบอย่างไร และมีกฎทางธุรกิจใดบ้างที่เกี่ยวข้อง แบบจำลองแนวคิดมักสร้างขึ้นโดยเป็นส่วนหนึ่งของกระบวนการรวบรวมความต้องการของโครงการในเบื้องต้น
* **Logical database schemas (สกีมาฐานข้อมูลเชิงตรรกะ):** มีความกึ่งนามธรรมน้อยกว่าเมื่อเทียบกับ Conceptual schemas โดยจะระบุวัตถุของสกีมาอย่างชัดเจนด้วยข้อมูล เช่น ชื่อตาราง ชื่อฟิลด์ ความสัมพันธ์ของเอนทิตี และข้อจำกัดความสมบูรณ์ (Integrity constraints)—เช่น กฎใดๆ ที่ควบคุมฐานข้อมูล อย่างไรก็ตาม โดยทั่วไปแล้วจะไม่รวมถึงข้อกำหนดทางเทคนิคใดๆ
* **Physical database schemas (สกีมาฐานข้อมูลระดับกายภาพ):** ให้ข้อมูลทางเทคนิคที่ประเภทของ Logical database schema ขาดไป นอกเหนือจากข้อมูลบริบท เช่น ชื่อตาราง ชื่อฟิลด์ ความสัมพันธ์ของเอนทิตี และอื่นๆ นั่นคือ รวมถึง Syntax ที่จะใช้ในการสร้างโครงสร้างข้อมูลเหล่านี้ภายในพื้นที่จัดเก็บข้อมูลบนดิสก์

### Star schema เทียบกับ Snowflake schema (เพิ่มเติม)
ทั้งใน Logical schemas และ Physical schemas ตารางฐานข้อมูลจะมีคีย์หลัก (Primary key) หรือคีย์นอก (Foreign key) ซึ่งทำหน้าที่เป็นตัวระบุเฉพาะสำหรับแต่ละรายการในตาราง คีย์เหล่านี้ใช้ในคำสั่ง SQL เพื่อรวมตารางเข้าด้วยกัน สร้างมุมมองข้อมูลที่เป็นหนึ่งเดียว

Star schema คือประเภทของสกีมาฐานข้อมูลเชิงสัมพันธ์ที่ประกอบด้วย Fact table กลางเพียงตารางเดียวที่ล้อมรอบด้วย Dimension tables สิ่งนี้มักถูกพิจารณาว่าเป็นสกีมาที่เรียบง่ายกว่าเมื่อเทียบกับ Snowflake schema

Snowflake schema ประกอบด้วย Fact table หนึ่งตารางที่เชื่อมต่อกับ Dimension tables หลายตาราง ซึ่งตารางเหล่านั้นสามารถเชื่อมต่อกับ Dimension tables อื่นๆ ผ่านความสัมพันธ์แบบกลุ่มต่อหนึ่ง (Many-to-one) สกีมานี้มีข้อดีคือมีระดับความซ้ำซ้อนของข้อมูลต่ำ แต่ไม่มีประสิทธิภาพเท่าใดนักในเรื่องของประสิทธิภาพการคิวรี

### ประโยชน์ของสกีมาฐานข้อมูล (เพิ่มเติม)
* **การเข้าถึงและความปลอดภัย (Access and security):** การออกแบบสกีมาฐานข้อมูลช่วยจัดระเบียบข้อมูลเป็นเอนทิตีที่แยกจากกัน ทำให้ง่ายต่อการแชร์สกีมาเดียวภายในฐานข้อมูลอื่น ผู้ดูแลระบบยังสามารถควบคุมการเข้าถึงผ่านการอนุญาตของฐานข้อมูล เพิ่มการรักษาความปลอดภัยอีกชั้นหนึ่งสำหรับข้อมูลที่เป็นกรรมสิทธิ์มากขึ้น ตัวอย่างเช่น สกีมาเดียวอาจมีข้อมูลที่ระบุตัวบุคคลได้ (PII) ซึ่งคุณต้องการเข้ารหัสเพื่อจุดประสงค์ด้านความเป็นส่วนตัวและความปลอดภัย
* **การจัดระเบียบและการสื่อสาร (Organization and communication):** เอกสารสกีมาฐานข้อมูลช่วยให้สามารถจัดระเบียบและสื่อสารได้ดีขึ้นระหว่างผู้มีส่วนได้ส่วนเสียภายใน เนื่องจากให้แหล่งความจริงเดียว (Single source of truth) จึงช่วยให้ผู้ใช้เข้าใจข้อจำกัดเชิงตรรกะและวิธีการรวมข้อมูลระหว่างตาราง
* **ความถูกต้องสมบูรณ์ (Integrity):** การจัดระเบียบและการสื่อสารนี้ยังช่วยรับประกันความถูกต้องของข้อมูลอีกด้วย ตัวอย่างเช่น สามารถช่วยผู้ดูแลระบบจัดการกระบวนการ Normalization เพื่อหลีกเลี่ยงความซ้ำซ้อนของข้อมูล นอกจากนี้ยังสามารถช่วยในการตรวจสอบการปฏิบัติตามข้อจำกัดในการออกแบบฐานข้อมูลของสกีมา ทำให้สามารถปฏิบัติตามคุณสมบัติ ACID ได้ (Atomicity, Consistency, Isolation, Durability)

# ส่วนที่ 2: NoSQL Database คืออะไร?

> 🗄️ **แหล่งที่มา:** [IBM — What Is a NoSQL Database?](https://www.ibm.com/think/topics/nosql-databases)

## 2.1 NoSQL คืออะไร?
**NoSQL (Not Only SQL หรือ Non-SQL)** คือแนวทางการออกแบบฐานข้อมูลที่ใช้จัดเก็บและดึงข้อมูลโดย **ไม่ใช้โครงสร้างแบบตาราง (Tabular)** เหมือนในฐานข้อมูลเชิงสัมพันธ์ (Relational Database) แบบดั้งเดิม
*   **Schema-less:** ไม่ต้องกำหนดโครงสร้างข้อมูลล่วงหน้าให้เป๊ะ ทำให้มีความยืดหยุ่นสูงมาก
*   **Distributed Database:** ข้อมูลถูกเก็บกระจายไว้ในหลายๆ เซิร์ฟเวอร์ (Distributed) ทำให้ทนทานต่อความเสียหาย (High Availability) ถ้าระบบบางส่วนดาวน์ ระบบที่เหลือก็ยังทำงานได้
*   **Scalability:** สามารถขยายระบบได้รวดเร็วเพื่อรองรับข้อมูลที่ไม่มีโครงสร้างชัดเจนจำนวนมหาศาล

---

## 2.2 NoSQL แตกต่างจาก SQL อย่างไร?
*   **SQL (Relational):** เก็บข้อมูลแบบมีตาราง มีความสัมพันธ์กัน (Joined tables) เหมาะกับข้อมูลที่มีแบบแผนชัดเจน แต่มีข้อจำกัดในการขยายระบบ เพราะส่วนใหญ่การขยายต้องเป็นวง **Vertical Scaling** (เพิ่มสเปค CPU/RAM ให้เซิร์ฟเวอร์ทื่อๆ ซึ่งมีขีดจำกัด)
*   **NoSQL (Non-relational):** กำจัดปัญหาการทับซ้อนของการ Join ตารางทิ้งไป และถูกออกแบบมาให้ทำ **Horizontal Scaling** (การเพิ่มจำนวนเซิร์ฟเวอร์ให้ช่วยกันทำงาน) ได้ง่ายและสมบูรณ์แบบกว่าตั้งแต่วันแรก เพื่อรองรับปริมาณการใช้งานที่มากขึ้น

## 2.3 ประเภทของฐานข้อมูล NoSQL (Types of NoSQL)
ฐานข้อมูล NoSQL มีแบ่งย่อยออกไปอีก 4-5 รูปแบบหลัก ตามความเหมาะสมของงาน:

1.  **Key-value store:** รูปแบบที่เรียบง่ายที่สุด เก็บข้อมูลเป็นคู่ Key และ Value (คล้าย Dictionary) เหมาะใช้สำหรับทำ Caching หรือ เก็บข้อมูลชั่วคราวอย่างตะกร้าสินค้าของ User (User Sessions)
    *   *ตัวอย่าง:* Redis, Memcached
2.  **Document store:** เก็บข้อมูลเป็นเอกสาร (เอกสาร 1 ชิ้น = 1 Record) มักเป็นฟอร์แมต JSON, XML ข้อมูลแต่ละชิ้นไม่ต้องมีฟิลด์เหมือนกันเป๊ะ ทำให้ยืดหยุ่นต่อการเพิ่มฟิลด์ใหม่ๆ ในอนาคต เหมาะกับระบบโปรไฟล์ หรือ Content Management (CMS)
    *   *ตัวอย่าง:* MongoDB
3.  **Wide-column store:** เก็บข้อมูลแยกในรูปแบบคอลัมน์ (Column-oriented) แทนที่จะเป็นแถว (Row-oriented) ระบบดึงเฉพาะคอลัมน์ที่ต้องการโดยไม่ต้องโหลดคอลัมน์อื่นมาให้หนักเครื่อง เหมาะกับงานวิเคราะห์ข้อมูลใหญ่ๆ (Big data analytics)
    *   *ตัวอย่าง:* Apache Cassandra, HBase
4.  **Graph store:** เก็บข้อมูลในรูปแบบจุด (Nodes) และเส้นความสัมพันธ์ (Edges) เหมาะสำหรับระบบที่ข้อมูลทุกอย่างเกี่ยวโยงกันเป็นตาข่ายซับซ้อน เช่น Social Network (ใครเป็นเพื่อนกับใคร), แนะนำสินค้า (Recommendation)
    *   *ตัวอย่าง:* Neo4j
5.  **In-memory store:** เก็บข้อมูลลงในหน่วยความจำหลัก (RAM) ของเซิร์ฟเวอร์ทั้งหมด แทนที่จะเขียนลงฮาร์ดดิสก์ ทำให้การดึงข้อมูลทำงานได้ความเร็วระดับสายฟ้าแลบ

## 2.4 ข้อดีหลักของ NoSQL (Advantages of NoSQL)
*   **Cost-effectiveness (ประหยัดค่าใช้จ่าย):** ขยายระบบแบบ Horizontal ด้วยฮาร์ดแวร์ธรรมดาๆ หลายเครื่องได้ ประหยัดกว่าการต้องซื้อ Database Server ระดับไฮเอนด์ตัวเดียว
*   **Flexibility (ยืดหยุ่น):** เมื่อไม่ต้องฟิกซ์ Schema ล่วงหน้า การปรับแก้ไขโค้ด อัปเดตฟีเจอร์ใหม่ๆ ก็ทำได้รวดเร็ว (Agile Development) ไม่มีสะดุด
*   **Replication (ทำสำเนาอัตโนมัติ):** ด้วยกลไกทำสำเนาไปเก็บหลายๆ เซิร์ฟเวอร์ ทำให้ระบบพร้อมใช้งานเสมอ และป้องกันข้อมูลสูญหาย
*   **Speed (ทำงานรวดเร็ว):** ตอบโจทย์ Web Application หรือ Mobile App สมัยใหม่ที่ความเร็วในการโหลดข้อมูลส่งผลต่อความพึงพอใจโดยตรง

## 2.5 Use Cases และ Microservices
ยุคนี้โปรเจกต์ขนาดใหญ่มักไม่ได้ใช้ฐานข้อมูลตัวเดียวคุมทั้งระบบ (Monolithic) แต่ใช้สถาปัตยกรรม **Microservices** (แยกฟีเจอร์เป็นเซิร์ฟวิสย่อยๆ)

ซึ่ง NoSQL ช่วยสนับสนุนแนวคิดที่ว่า **"ใช้ฐานข้อมูลที่เหมาะสมที่สุดสำหรับงานนั้นๆ" (Polyglot Persistence)** เช่น ระบบตะกร้าสินค้าใช้ Redis (Key-value), ส่วนการเก็บประวัติแชทใช้ MongoDB (Document) ทำให้ไม่ต้องพยายามฝืนเอาทุกอย่างไปยัดไว้ใน SQL ตัวเดียวเหมือนในอดีต

## 2.6 เนื้อหาเพิ่มเติม: NoSQL Database รายละเอียดเชิงลึก

### NoSQL Database คืออะไร? (ฉบับเต็ม)
NoSQL ย่อมาจาก "not only SQL" หรือ "non-SQL" คือแนวทางการออกแบบฐานข้อมูลที่ช่วยให้สามารถจัดเก็บและเรียกค้นข้อมูลได้ภายนอกโครงสร้างแบบดั้งเดิมที่พบในฐานข้อมูลเชิงสัมพันธ์ (Relational Databases)

แม้ว่า NoSQL จะยังคงสามารถจัดเก็บข้อมูลที่พบในระบบจัดการฐานข้อมูลเชิงสัมพันธ์ (RDBMS) ได้ แต่จะจัดเก็บแตกต่างกันเมื่อเทียบกับ RDBMS การตัดสินใจเลือกระหว่างการใช้ฐานข้อมูลเชิงสัมพันธ์กับฐานข้อมูลที่ไม่ใช่เชิงสัมพันธ์นั้นขึ้นอยู่กับบริบทและแตกต่างกันไปตามกรณีการใช้งาน

แทนที่จะใช้โครงสร้างตารางที่เป็นแบบฉบับของฐานข้อมูลเชิงสัมพันธ์ ฐานข้อมูล NoSQL จะเก็บข้อมูลไว้ในโครงสร้างข้อมูลเดียว เช่น เอกสาร JSON เนื่องจากสถาปัตยกรรมฐานข้อมูลที่ไม่ใช่เชิงสัมพันธ์นี้ไม่จำเป็นต้องใช้สกีมา (Schema) จึงทำให้สามารถขยายระบบได้อย่างรวดเร็วเพื่อจัดการชุดข้อมูลขนาดใหญ่และข้อมูลที่ไม่มีโครงสร้างโดยทั่วไป

นอกจากนี้ NoSQL ยังเป็นฐานข้อมูลประเภทกระจายศูนย์ (Distributed Database) ซึ่งหมายความว่าข้อมูลจะถูกคัดลอกและจัดเก็บไว้บนเซิร์ฟเวอร์หลายเครื่อง ซึ่งอาจเป็นระบบระยะไกล (Remote) หรือภายในพื้นที่ (Local) สิ่งนี้ช่วยรับประกันความพร้อมใช้งานและความน่าเชื่อถือของข้อมูล หากข้อมูลบางส่วนออฟไลน์ไป ส่วนที่เหลือของฐานข้อมูลก็ยังสามารถทำงานต่อไปได้

### NoSQL เทียบกับ SQL (ฉบับเต็ม)
ภาษาคิวรีมาตรฐาน (SQL) มักถูกอ้างอิงถึงเมื่อพูดถึง NoSQL ก่อนที่จะมีฐานข้อมูลเชิงสัมพันธ์ บริษัทต่างๆ ใช้ระบบฐานข้อมูลแบบลำดับชั้น (Hierarchical database system) ที่มีโครงสร้างคล้ายต้นไม้สำหรับตารางข้อมูล ระบบจัดการฐานข้อมูล (DBMS) ยุคแรกๆ เหล่านี้ช่วยให้ผู้ใช้จัดระเบียบข้อมูลปริมาณมากได้ อย่างไรก็ตาม มีความซับซ้อน มักเป็นกรรมสิทธิ์ของแอปพลิเคชันใดแอปพลิเคชันหนึ่ง และมีข้อจำกัดในวิธีการค้นหาข้อมูลภายใน ข้อจำกัดเหล่านี้ส่งผลให้เกิดการพัฒนาระบบจัดการฐานข้อมูลเชิงสัมพันธ์ในที่สุด

เมื่อเวลาผ่านไป ความต้องการการใช้งานชุดข้อมูลขนาดใหญ่ที่รวดเร็วขึ้นและแยกจากกันมากขึ้น มีความสำคัญเพิ่มขึ้นสำหรับเทคโนโลยีที่เกิดขึ้นใหม่ โปรแกรมเมอร์ต้องการสิ่งที่ยืดหยุ่นกว่าฐานข้อมูล SQL NoSQL จึงกลายเป็นทางเลือกนั้น

แม้ว่า NoSQL จะเป็นทางเลือกแทน SQL แต่ความก้าวหน้านี้ไม่ได้เข้ามาแทนที่ฐานข้อมูล SQL แต่อย่างใด บริษัทต่างๆ มักจะใช้ทั้ง NoSQL และฐานข้อมูลเชิงสัมพันธ์ร่วมกันในแอปพลิเคชันเดียว

ฐานข้อมูล SQL สามารถขยายได้เฉพาะในแนวตั้ง (Vertically) ไม่ใช่ในแนวนอน (Horizontally) ในทางเปรียบเทียบ ฐานข้อมูล NoSQL ไม่ใช่เชิงสัมพันธ์ ซึ่งช่วยขจัดความจำเป็นในการเชื่อมต่อตาราง ความสามารถในการทำ Sharding และความพร้อมใช้งานสูงที่มีมาในตัวช่วยให้การขยายขนาดในแนวนอนทำได้ง่ายขึ้น

### ประเภทของฐานข้อมูล NoSQL (ฉบับเต็ม)

**Key-value store (ตัวเก็บข้อมูลแบบกุญแจและค่า)**
โดยทั่วไปถือว่าเป็นรูปแบบที่ง่ายที่สุดของฐานข้อมูล NoSQL โมเดลข้อมูลที่ไม่มีสกีมานี้จัดระเบียบเป็นพจนานุกรมของคู่คีย์-ค่า (Key-value pairs) มักใช้สำหรับการทำแคช (Caching) และการจัดเก็บข้อมูลเซสชันของผู้ใช้ Redis และ Memcached เป็นตัวอย่างของฐานข้อมูลคีย์-ค่าแบบโอเพนซอร์ส

**Document store (ตัวเก็บข้อมูลแบบเอกสาร)**
ฐานข้อมูลเอกสารจะเก็บข้อมูลเป็นเอกสาร ช่วยในการจัดการข้อมูลแบบกึ่งโครงสร้าง (Semi-structured data) และข้อมูลมักจะถูกจัดเก็บในรูปแบบ JSON, XML หรือ BSON กรณีการใช้งานที่นิยม ได้แก่ ระบบจัดการเนื้อหา (CMS) และโปรไฟล์ผู้ใช้ ตัวอย่างคือ MongoDB

**Wide-column store (ตัวเก็บข้อมูลแบบคอลัมน์กว้าง)**
ฐานข้อมูลเหล่านี้จัดเก็บข้อมูลเป็นคอลัมน์ ช่วยให้ผู้ใช้สามารถเข้าถึงเฉพาะคอลัมน์ที่ต้องการโดยไม่ต้องจัดสรรหน่วยความจำเพิ่มเติมให้กับข้อมูลที่ไม่เกี่ยวข้อง Apache HBase และ Apache Cassandra เป็นตัวอย่างของฐานข้อมูลแบบ Wide-column โอเพนซอร์ส

**Graph store (ตัวเก็บข้อมูลแบบกราฟ)**
ฐานข้อมูลประเภทนี้มักใช้เก็บข้อมูลจากกราฟความรู้ (Knowledge graph) องค์ประกอบของข้อมูลจะถูกจัดเก็บเป็นโหนด (Nodes), เส้นเชื่อม (Edges) และคุณสมบัติ (Properties) Neo4j เป็นบริการฐานข้อมูลแบบกราฟที่ใช้ภาษา Java

**In-memory store (ตัวเก็บข้อมูลในหน่วยความจำ)**
ข้อมูลจะอยู่ในหน่วยความจำหลัก (RAM) แทนที่จะอยู่บนดิสก์ ทำให้เข้าถึงข้อมูลได้เร็วกว่าฐานข้อมูลบนดิสก์แบบเดิม

### ตัวอย่างของฐานข้อมูล NoSQL
* Apache CouchDB ฐานข้อมูลแบบเอกสาร JSON โอเพนซอร์สที่ใช้ภาษา JavaScript เป็นภาษาคิวรี
* Elasticsearch ฐานข้อมูลแบบเอกสารที่มีเอ็นจิ้นการค้นหาข้อความแบบเต็ม (Full-text search engine)
* Couchbase ฐานข้อมูลแบบคีย์-ค่าและเอกสารที่ช่วยให้นักพัฒนาสร้างแอปพลิเคชันที่ตอบสนองและยืดหยุ่น

### กรณีการใช้งาน NoSQL (NoSQL use cases)
* **การจัดการความสัมพันธ์ของข้อมูล:** ใช้ฐานข้อมูล NoSQL แบบกราฟ ซึ่งรวมถึงเอ็นจิ้นการแนะนำ กราฟความรู้ แอปพลิเคชันตรวจจับการฉ้อโกง และเครือข่ายโซเชียล
* **ประสิทธิภาพความหน่วงต่ำ (Low-latency performance):** การเล่นเกม แอปพลิเคชันฟิตเนส และเทคโนโลยีโฆษณา ต้องการฐานข้อมูล NoSQL แบบ In-memory เพื่อให้เวลาตอบสนองรวดเร็ว
* **การขยายขนาดและปริมาณข้อมูลขนาดใหญ่:** อีคอมเมิร์ซต้องการความสามารถในการจัดการกับการใช้งานที่พุ่งสูงขึ้น ฐานข้อมูลแบบคีย์-ค่ามักใช้เพราะโครงสร้างที่เรียบง่ายสามารถขยายขนาดได้ง่าย

### Microservices และฐานข้อมูล NoSQL (เพิ่มเติม)
ไมโครเซอร์วิสเป็นตัวเลือกที่น่าสนใจ ส่วนหนึ่งเป็นเพราะช่วยลดความจำเป็นในการใช้ที่เก็บข้อมูลเดียวที่แชร์ร่วมกันสำหรับแอปพลิเคชันทั้งหมด แต่ละบริการมีโมเดลข้อมูลและฐานข้อมูลของตัวเอง และรวมเข้าด้วยกันผ่านเกตเวย์ API หรือ iPaaS

รูปแบบของการใช้ฐานข้อมูลหลายชุดภายในแอปพลิเคชันเดียว ซึ่งเรียกว่า Polyglot Persistence ได้ช่วยสร้างพื้นที่ในตลาดให้ฐานข้อมูล NoSQL เจริญเติบโต

# ส่วนที่ 3: CAP Theorem คืออะไร?

> ⚖️ **แหล่งที่มา:** [IBM — What Is the CAP Theorem?](https://www.ibm.com/think/topics/cap-theorem)

## 3.1 CAP Theorem คืออะไร?
**CAP Theorem (หรือ Brewer's Theorem)** เป็นกฎเหล็กของการออกแบบระบบแบบกระจายศูนย์ (Distributed Systems) ที่กล่าวไว้ว่า:
> **"ระบบ Distributed System สามารถเลือกคุณสมบัติที่ดีที่สุดได้เพียง 2 ใน 3 อย่างนี้เท่านั้น (Consistency, Availability, หรือ Partition tolerance)"**

เปรียบเทียบง่ายๆ เหมือนเวลารับเหมาก่อสร้างที่มักจะมีกฎว่า "ถูก, เร็ว, หรือ ดี — คุณเลือกได้แค่ 2 อย่าง"

---

## 3.2 ตัวอักษร C, A, และ P หมายถึงอะไร?

### 🪞 C: Consistency (ความสม่ำเสมอของข้อมูล)
หมายความว่า **"Client ทุกคนต้องเห็นข้อมูลล่าสุดที่ตรงกันเสมอ ไม่ว่าจะดึงข้อมูลมาจาก Node ไหนก็ตาม"**
*   **เงื่อนไข:** เมื่อมีการเขียนข้อมูลลง Node หนึ่ง ข้อมูลนั้นต้องถูกคัดลอก (Replicate) ไปยัง Node อื่นๆ ในระบบทั้งหมดให้เสร็จสิ้นก่อน จึงจะนับว่าการเขียน "สำเร็จ"

### 🟢 A: Availability (ความพร้อมใช้งาน)
หมายความว่า **"ทุกคำขอ (Request) ต้องได้รับการตอบกลับเสมอ แม้ว่าจะมีบาง Node ในระบบพังอยู่ก็ตาม"**
*   **เงื่อนไข:** Node ที่ยังรอดชีวิตอยู่ ต้องตอบกลับข้อมูลได้ทุกครั้งแบบไม่มีข้อยกเว้น ไม่มีการปล่อยให้ระบบค้างหรือขึ้น Error ว่าไม่พร้อมใช้งาน

### ✂️ P: Partition tolerance (ความทนทานต่อการตัดขาดของเครือข่าย)
"Partition" คือสถานการณ์ที่เครือข่ายระหว่าง 2 Nodes ถูกตัดขาดหรือคุยกันไม่ได้ชั่วคราว (Network break)
*   **เงื่อนไข:** ระบบโดยรวมต้อง **ยังคงทำงานต่อไปได้** แม้ว่าจะเกิดการตัดขาดการเชื่อมต่อระหว่าง Node จำนวนเท่าใดก็ตาม

---

## 3.3 รูปแบบฐานข้อมูลตามทฤษฎี CAP

ในโลกของระบบกระจายศูนย์ (Distributed Systems) อาการเครือข่ายหลุด (Partition) เป็นสิ่งที่ "หลีกเลี่ยงไม่ได้" ดังนั้นเราจึง**ต้องรวม P ไว้เสมอ** หมายความว่าในการเลือกฐานข้อมูล NoSQL เราจึงเลือกได้แค่ **CP** หรือ **AP** เท่านั้น:

1.  **CP (Consistency + Partition tolerance):**
    *   *เน้นความถูกต้อง:* ยอมทิ้งความพร้อมใช้งาน (Availability) ไป หากระบบเจอ Network Partition Node ที่มีปัญหาจะถูกสั่งให้ "หยุดให้บริการชั่วคราว" จนกว่าเครือข่ายจะกลับมา เพื่อป้องกันปัญหาข้อมูลไม่ตรงกัน
    *   *ตัวอย่าง:* **MongoDB** (เป็นระบบมี Master เดียว ถ้า Master ล่ม จะหยุดรับข้อมูลชั่วคราวจนกว่าจะเลือก Master ใหม่เสร็จ)
2.  **AP (Availability + Partition tolerance):**
    *   *เน้นพร้อมใช้งาน:* ยอมทิ้งความสม่ำเสมอ (Consistency) ไป ทุก Node จะยังคงรับแขกและคืนข้อมูลให้ Client เสมอ อาจทำให้บางคนได้ข้อมูลเก่าไปบ้าง (Stale data) แต่พอเน็ตเวิร์กกลับมาปกติ ระบบจะทำการซิงค์ข้อมูลให้ตรงกันในภายหลัง (เรียกว่า **Eventual Consistency**)
    *   *ตัวอย่าง:* **Apache Cassandra**, CouchDB
3.  **CA (Consistency + Availability):**
    *   *ตัวละครลับ:* ถ้าจะเอาทั้งตรงกันและพร้อมเสมอ จะไม่สามารถทนทานต่อเครือข่ายขาดได้เลย ซึ่งในโลกความเป็นจริงเป็นไปไม่ได้ใน Distributed System อย่างไรก็ตาม ฐานข้อมูลเชิงสัมพันธ์แบบเดี่ยวๆ เช่น **PostgreSQL** มักถูกจัดในกลุ่มนี้

---

## 3.4 สรุป: จะเลือกใช้อะไรตอนออกแบบ Microservices?

การเข้าใจ CAP Theorem ทำให้คุณเลือกเครื่องมือได้ตรงกับโดเมนของธุรกิจ:
*   ถ้าทำแอประบบตะกร้าสินค้า โซเชียลมีเดีย หรือฟีดข่าว ที่ต้องการให้แอปไม่ล่มและโหลดเร็ว (รับได้ถ้าคอมเมนต์เพื่อนอัปเดตช้าไป 2 วิ) 👉 **ควรเลือก AP (เช่น Cassandra)**
*   ถ้าทำแอประบบธนาคาร ตัดเงิน ยอดเงินในบัญชี (ที่การเห็นข้อมูลผิดไป 10 วินาทีคือหายนะ) 👉 **ต้องเน้น Consistency แบบ CP หรือใช้ Relational DB อย่าง PostgreSQL แทน**

---

## 3.5 เนื้อหาเพิ่มเติม: CAP Theorem รายละเอียดเชิงลึก

### ประวัติของ CAP Theorem
ทฤษฎี CAP มีชื่อเรียกอีกอย่างว่า ทฤษฎีบทของบริวเวอร์ (Brewer's Theorem) เนื่องจากศาสตราจารย์ Eric A. Brewer เป็นผู้เสนอขึ้นเป็นครั้งแรกในระหว่างการบรรยายเกี่ยวกับการประมวลผลแบบกระจายศูนย์ในปี 2000 สองปีต่อมา ศาสตราจารย์ Seth Gilbert และ Nancy Lynch จาก MIT ได้ตีพิมพ์บทพิสูจน์ของ "Brewer's Conjecture"

### ประเภทฐานข้อมูล NoSQL ตามทฤษฎี CAP (ฉบับเต็ม)

**CP database:** ฐานข้อมูล CP ส่งมอบความสอดคล้องและความทนทานต่อการตัดขาดของเครือข่ายโดยแลกกับความพร้อมใช้งาน เมื่อเกิด Partition ระหว่างสองโหนดใดๆ ระบบจะต้องปิดโหนดที่ไม่สอดคล้องกัน จนกว่า Partition จะได้รับการแก้ไข

**AP database:** ฐานข้อมูล AP ส่งมอบความพร้อมใช้งานและความทนทานต่อการตัดขาดของเครือข่ายโดยแลกกับความสอดคล้อง เมื่อเกิด Partition โหนดทั้งหมดจะยังคงพร้อมใช้งาน แต่โหนดบางตัวอาจส่งกลับข้อมูลเวอร์ชันเก่ากว่า (เมื่อ Partition ได้รับการแก้ไขแล้ว ฐานข้อมูล AP จะทำการซิงค์โหนดใหม่เพื่อซ่อมแซมความไม่สอดคล้องกัน)

**CA database:** ฐานข้อมูล CA ส่งมอบความสอดคล้องและความพร้อมใช้งานในทุกโหนด อย่างไรก็ตาม มันไม่สามารถทำได้หากมี Partition ในระบบ ดังนั้นจึงไม่สามารถส่งมอบความทนทานต่อความผิดพลาดได้ ในทางปฏิบัติ ฐานข้อมูลแบบกระจาย CA ไม่สามารถดำรงอยู่ได้ แต่ฐานข้อมูลเชิงสัมพันธ์หลายระบบ เช่น PostgreSQL มอบความสอดคล้องและความพร้อมใช้งาน และสามารถปรับใช้กับหลายโหนดได้โดยการทำซ้ำ (Replication)

### MongoDB และทฤษฎี CAP (CP)
MongoDB เป็นระบบจัดการฐานข้อมูล NoSQL ยอดนิยมที่จัดเก็บข้อมูลเป็นเอกสาร BSON (Binary JSON) เมื่อเปรียบเทียบกับทฤษฎี CAP แล้ว MongoDB เป็นที่เก็บข้อมูลแบบ CP นั่นคือ มันแก้ปัญหา Network Partition โดยการรักษาความสอดคล้อง ในขณะที่ประนีประนอมกับความพร้อมใช้งาน

MongoDB เป็นระบบที่มีมาสเตอร์เพียงตัวเดียว (Single-master system) แต่ละชุด Replica set สามารถมีโหนดหลัก (Primary node) ได้เพียงโหนดเดียวที่รับการทำงานการเขียนทั้งหมด โหนดอื่นๆ ทั้งหมดคือโหนดรอง (Secondary nodes) ที่ทำซ้ำบันทึกการทำงานของโหนดหลัก

เมื่อโหนดหลักไม่สามารถใช้งานได้ โหนดรองที่มีบันทึกการทำงานล่าสุดจะถูกเลือกเป็นโหนดหลักใหม่ เนื่องจากไคลเอนต์ไม่สามารถทำการร้องขอการเขียนใดๆ ได้ในช่วงเวลาดังกล่าว ข้อมูลจึงยังคงสอดคล้องกันทั่วทั้งเครือข่าย

### Cassandra และทฤษฎี CAP (AP)
Apache Cassandra เป็นฐานข้อมูล NoSQL แบบโอเพนซอร์สแบบ Wide-column ที่ให้คุณจัดเก็บข้อมูลบนเครือข่ายแบบกระจาย Cassandra มีสถาปัตยกรรมแบบไม่มีมาสเตอร์ (Masterless architecture) เมื่อเปรียบเทียบกับทฤษฎี CAP แล้ว Cassandra เป็นฐานข้อมูลแบบ AP

Cassandra มอบความสอดคล้องในท้ายที่สุด (Eventual consistency) โดยอนุญาตให้ไคลเอนต์เขียนไปยังโหนดใดก็ได้ตลอดเวลา และประสานความไม่สอดคล้องกันให้เร็วที่สุดเท่าที่จะเป็นไปได้ Cassandra เสนอฟังก์ชัน "ซ่อมแซม" เพื่อช่วยให้โหนดไล่ตามทันโหนดอื่นๆ

### ไมโครเซอร์วิสและทฤษฎี CAP
การทำความเข้าใจทฤษฎี CAP สามารถช่วยให้คุณเลือกฐานข้อมูลที่ดีที่สุดเมื่อออกแบบแอปพลิเคชันที่ใช้ไมโครเซอร์วิส:
*   หากความสามารถในการขยายขนาดในแนวนอนอย่างรวดเร็วเป็นสิ่งสำคัญ และคุณสามารถทนต่อ Eventual consistency ได้ 👉 ฐานข้อมูล AP เช่น Cassandra หรือ Apache CouchDB
*   หากแอปพลิเคชันของคุณพึ่งพาความสอดคล้องของข้อมูลเป็นอย่างมาก เช่น ในแอปพลิเคชันอีคอมเมิร์ซหรือบริการชำระเงิน 👉 ฐานข้อมูลเชิงสัมพันธ์ เช่น PostgreSQL

# ส่วนที่ 4: etcd คืออะไร?

> 🗄️ **แหล่งที่มา:** IBM — etcd

## 4.1 etcd คืออะไร?
etcd (อ่านว่า "เอ็ต-ซี-ดี") คือคลังเก็บข้อมูลแบบคีย์-ค่า (Key-value store) ที่มีความสอดคล้องกันและพร้อมใช้งานสูงแบบกระจายศูนย์ (Distributed, consistent, and highly available) ใช้เพื่อจัดเก็บข้อมูลการกำหนดค่า (Configuration), ข้อมูลเมทาดาตา (Metadata) และข้อมูลสถานะ (State) ของแอปพลิเคชันระบบแบบกระจายศูนย์ขนาดใหญ่

กรณีการใช้งานที่มีชื่อเสียงที่สุดคือการเป็นที่เก็บข้อมูลหลักเบื้องหลังสำหรับ Kubernetes แพลตฟอร์มการจัดการออร์เคสเตรชันคอนเทนเนอร์ (Container orchestration)

---

## 4.2 คุณสมบัติเด่นของ etcd
* **ทำสำเนาอย่างสมบูรณ์ (Fully Replicated):** ทุกโหนดในคลัสเตอร์ etcd สามารถเข้าถึงคลังข้อมูลทั้งหมดได้
* **พร้อมใช้งานสูง (Highly Available):** ถูกออกแบบมาเพื่อหลีกเลี่ยงจุดล้มเหลวจุดเดียว (Single point of failure) และทนทานต่อความล้มเหลวของฮาร์ดแวร์และเครือข่าย
* **มีความสอดคล้องที่เชื่อถือได้ (Reliably Consistent):** ข้อมูลที่ถูกอ่านจากคลัสเตอร์ในทุกจุดจะเป็นข้อมูลล่าสุดที่ถูกเขียนเสมอ
* **รวดเร็ว (Fast):** ผ่านการทดสอบเกณฑ์มาตรฐานที่รองรับการเขียนข้อมูลมากกว่า 10,000 ครั้งต่อวินาที
* **ปลอดภัย (Secure):** รองรับระบบรักษาความปลอดภัย TLS อัตโนมัติ และการตรวจสอบสิทธิ์ผ่าน Client Certificate
* **ใช้งานง่าย (Simple):** แอปพลิเคชันทั่วไปสามารถสื่อสารผ่าน HTTP/JSON หรือ gRPC มาตรฐานได้

---

## 4.3 etcd และ Kubernetes
ในสถาปัตยกรรมของ Kubernetes etcd ทำหน้าที่เป็นหน่วยเก็บข้อมูลหลักสำหรับ Control plane

เมื่อผู้ใช้สั่งสร้าง Pod หรืออัปเดตสถานะการทำงาน Kubernetes API Server จะบันทึกสถานะที่ต้องการ (Desired state) ลงใน etcd ตัว API Server จะคอยเปรียบเทียบสถานะจริงจาก Nodes เทียบกับสถานะที่บันทึกไว้ใน etcd เพื่อให้แน่ใจว่าระบบทำงานได้ตามความต้องการของผู้ใช้อยู่ตลอดเวลา

เนื่องจาก Kubernetes พึ่งพา etcd อย่างหนักในการทำซิงโครไนซ์สถานะ IBM จึงแนะนำให้รัน etcd บนไดรฟ์ประเภท SSD (Solid-State Drives) เพื่อเพิ่มประสิทธิภาพการรับส่งข้อมูล (I/O) ที่มีความหน่วงต่ำ

---

## 4.4 อัลกอริทึม Raft Consensus
etcd บรรลุความสอดคล้องของข้อมูลผ่านการใช้อัลกอริทึมการลงมติพ้องต้องกัน (Consensus) ที่ชื่อว่า Raft Algorithm

* Raft ทำงานโดยการเลือกโหนดหนึ่งขึ้นมาทำหน้าที่เป็น **ผู้นำ (Leader)** และโหนดอื่น ๆ เป็น **ผู้ตาม (Followers)**
* คำสั่งเขียนข้อมูลทั้งหมดจะต้องส่งผ่านโหนด Leader
* เมื่อมีคำสั่งเขียน Leader จะส่งข้อมูลไปยัง Followers และรอเสียงโหวตยอมรับจากเสียงส่วนใหญ่ของกลุ่ม (Quorum) เช่น 3 จาก 5 โหนด
* เมื่อได้เสียงส่วนใหญ่แล้ว ข้อมูลจึงจะถูก Commit ลงจัดเก็บถาวรและแจ้งให้เครื่องอื่น ๆ บันทึกตาม
* หาก Leader ล้มเหลวไป ระบบจะเปิดโหวตเลือก Leader คนใหม่ขึ้นมาทันทีโดยอัตโนมัติ ทำให้ระบบสามารถทำงานต่อได้โดยไม่หยุดชะงัก

# ส่วนที่ 5: Relational Database คืออะไร?

> 🗃️ **แหล่งที่มา:** IBM — Relational Databases

## 5.1 Relational Database คืออะไร?
ฐานข้อมูลเชิงสัมพันธ์จัดเก็บและจัดระเบียบข้อมูลในรูปแบบของตาราง (Tables) ซึ่งประกอบด้วยแถวและคอลัมน์ จุดเด่นของฐานข้อมูลประเภทนี้คือความสามารถในการเชื่อมโยงข้อมูลระหว่างตารางเข้าด้วยกันผ่านค่าที่เรียกว่า คีย์ (Key) ทำให้สามารถดึงข้อมูลที่เกี่ยวข้องกันจากหลายตารางมาแสดงผลรวมกันได้อย่างแม่นยำ

---

## 5.2 องค์ประกอบสำคัญของ Relational Database
* **Primary Key (คีย์หลัก):** ฟิลด์ที่ใช้ระบุตัวตนของแต่ละแถวข้อมูลในตารางนั้น ๆ แบบไม่ซ้ำกัน เช่น รหัสพนักงาน
* **Foreign Key (คีย์นอก):** ฟิลด์ในตารางหนึ่งที่อ้างอิงไปยัง Primary Key ของอีกตารางหนึ่ง เพื่อสร้างความสัมพันธ์เชื่อมโยง

---

## 5.3 คุณสมบัติ ACID Properties
ฐานข้อมูลเชิงสัมพันธ์ทำงานภายใต้มาตรฐานที่เรียกว่า ACID เพื่อรับประกันความถูกต้องของการทำทรานแซกชัน:
* **A - Atomicity (ความเป็นหนึ่งเดียว):** ทรานแซกชันต้องทำงานสำเร็จทั้งหมด หรือไม่สำเร็จเลย หากล้มเหลวกลางคัน ระบบจะทำการย้อนคืนกลับ (Rollback) ข้อมูลไม่ตกค้าง
* **C - Consistency (ความสอดคล้อง):** ข้อมูลก่อนและหลังทำทรานแซกชันต้องถูกต้องตามกฎเกณฑ์ของสกีมาฐานข้อมูล
* **I - Isolation (ความเป็นอิสระ):** ทรานแซกชันที่ทำงานพร้อมกันต้องไม่รบกวนกันเอง
* **D - Durability (ความคงทน):** เมื่อระบบทำการบันทึกสำเร็จ ข้อมูลจะถูกบันทึกถาวรลงดิสก์ แม้ระบบล่มหลังจากนั้น ข้อมูลก็จะไม่หายไป

# ส่วนที่ 6: ถอดคำพูดจากวิดีโอ — What is a Relational Database? (IBM Technology)

> 🎥 **แหล่งที่มา:** IBM Technology YouTube — What is a Relational Database?

## 6.1 โครงสร้างของฐานข้อมูลเชิงสัมพันธ์
ฐานข้อมูลเชิงสัมพันธ์ทำงานโดยการสร้าง **ตาราง (Tables)** ซึ่งแสดงถึงเอนทิตีเฉพาะ (เช่น หนังสือ, ผู้เขียน) แต่ละตารางจะมี:
*   **คอลัมน์ (Columns):** แสดงคุณลักษณะ (Attributes) เช่น ชื่อ, วันที่, ผู้เขียน
*   **ฟิลด์ ID:** ใช้ระบุตัวตนแต่ละแถวอย่างไม่ซ้ำกัน
*   **แถว (Rows/Records):** แสดงถึงระเบียนข้อมูลจริง

การคิวรีข้อมูลทำผ่าน **SQL (Structured Query Language)**

---

## 6.2 ความสัมพันธ์ระหว่างตาราง
พลังของฐานข้อมูลเชิงสัมพันธ์อยู่ที่ความสามารถในการเชื่อมโยงเอนทิตีเข้าด้วยกัน:
*   **One-to-one:** ความสัมพันธ์แบบหนึ่งต่อหนึ่ง
*   **One-to-many:** ความสัมพันธ์แบบหนึ่งต่อกลุ่ม (เช่น หนังสือมีผู้เขียนได้หลายคน)
*   **Foreign Key:** ใช้เชื่อมโยงตารางเข้าด้วยกัน โดยอ้างอิง ID ของอีกตาราง

---

## 6.3 Index (ดัชนี)
เมื่อมีการรันคิวรีจำนวนมาก สามารถตั้งค่า **Index** บนคอลัมน์ที่ถูกค้นหาบ่อยๆ (เช่น ชื่อ หรือ วันที่) ซึ่งจะจัดทำดัชนีค่าทั้งหมดโดยอัตโนมัติ ทำให้การดึงข้อมูลครั้งถัดไปรวดเร็วขึ้นมาก ทุกครั้งที่มีการเพิ่มแถวใหม่ Index จะอัปเดตแคชของค่าเหล่านี้ให้เป็นปัจจุบันอยู่เสมอ

---

## 6.4 ประโยชน์ของฐานข้อมูลเชิงสัมพันธ์ (จากวิดีโอ)

### Consistency (ความสอดคล้อง)
หากต้องรันธุรกรรมหลายรายการติดต่อกัน และคิวรีใดคิวรีหนึ่งล้มเหลว สามารถ **Roll back** ชุดคิวรีทั้งหมดเพื่อรักษาความสอดคล้องของข้อมูล

### Stored Procedures
ความสามารถในการเขียนบล็อกหรือฟังก์ชันของโค้ด SQL หลายๆ บล็อกเข้าด้วยกัน ลดความซ้ำซ้อนของโค้ด SQL และเพิ่มประสิทธิภาพ นอกจากนี้ยังมี **Views** ที่ช่วยสร้างมุมมองข้อมูลที่กำหนดเองได้

### Locking และ Concurrency
กลไกการจัดการการล็อกในฐานข้อมูลเมื่อมีผู้ใช้หลายคนทำงานพร้อมกัน เมื่อแอปพลิเคชันแรกทำการแทรก (Insert) ข้อมูล ฐานข้อมูลจะถูกล็อกจนกว่าการเขียนจะเสร็จสิ้น แล้วจึงอนุญาตให้แอปพลิเคชันอื่นเขียนข้อมูลต่อได้ ความสามารถเหล่านี้มาพร้อมกับฐานข้อมูลเชิงสัมพันธ์โดยไม่ต้องเขียนโค้ดเพิ่มเติม
