## บทนำ

เอกสารนี้อธิบายโครงสร้างพื้นฐานและเทคโนโลยีที่ทำให้อินเทอร์เน็ตทำงานได้ โดยไม่ได้ลงลึกในรายละเอียดมากนัก แต่ครอบคลุมเนื้อหาในแต่ละส่วนเพียงพอที่จะให้ความเข้าใจพื้นฐานเกี่ยวกับแนวคิดที่เกี่ยวข้อง สำหรับคำถามที่ยังไม่ได้รับคำตอบ จะมีรายการแหล่งข้อมูลอยู่ท้ายเอกสาร

## เริ่มต้นจากตรงไหน? Internet Address

ภาพด้านล่างแสดงคอมพิวเตอร์สองเครื่องที่เชื่อมต่อกับอินเทอร์เน็ต โดยคอมพิวเตอร์ของคุณมี IP Address 1.2.3.4 และคอมพิวเตอร์อีกเครื่องมี IP Address 5.6.7.8 อินเทอร์เน็ตถูกแสดงเป็นวัตถุนามธรรมอยู่ตรงกลาง (เมื่อเอกสารนี้ดำเนินต่อไป ส่วนอินเทอร์เน็ตของแผนภาพ 1 จะถูกอธิบายและวาดใหม่หลายครั้งเมื่อรายละเอียดของอินเทอร์เน็ตถูกเปิดเผย)

> **ลองดู - โปรแกรม Ping**
>
> หากคุณใช้ Microsoft Windows หรือ Unix และมีการเชื่อมต่ออินเทอร์เน็ต มีโปรแกรมที่มีประโยชน์เพื่อดูว่าคอมพิวเตอร์บนอินเทอร์เน็ตยังทำงานอยู่หรือไม่ โปรแกรมนี้เรียกว่า **ping** ซึ่งน่าจะตั้งชื่อตามเสียงที่ระบบโซนาร์เรือดำน้ำรุ่นเก่าส่งออกมา
>
> หากคุณใช้ Windows ให้เปิดหน้าต่าง Command Prompt หากใช้ Unix ให้ไปที่ Command Prompt แล้วพิมพ์ `ping www.yahoo.com` โปรแกรม ping จะส่ง 'ping' (จริงๆ แล้วคือข้อความ ICMP (Internet Control Message Protocol) echo request) ไปยังคอมพิวเตอร์ที่ระบุชื่อ คอมพิวเตอร์ที่ถูก ping จะตอบกลับ โปรแกรม ping จะนับเวลาที่ใช้จนกว่าจะได้รับคำตอบกลับ (ถ้าได้รับ) นอกจากนี้ ถ้าคุณป้อน Domain Name (เช่น www.yahoo.com) แทนที่จะเป็น IP Address โปรแกรม ping จะแปลง Domain Name และแสดง IP Address ของคอมพิวเตอร์นั้น เรื่อง Domain Name และการแปลงที่อยู่จะกล่าวถึงในภายหลัง

## Protocol Stack และ Packet

เมื่อคอมพิวเตอร์ของคุณเชื่อมต่ออินเทอร์เน็ต ข้อมูลจะถูกส่งผ่าน **Protocol Stack** ดังนี้:

1. ข้อความจะเริ่มต้นที่ด้านบนของ Protocol Stack บนคอมพิวเตอร์ของคุณและเดินทางลงมาด้านล่าง

2. ถ้าข้อความที่จะส่งมีขนาดยาว แต่ละชั้นของ Stack ที่ข้อความผ่านอาจแบ่งข้อความออกเป็นก้อนข้อมูลขนาดเล็กกว่า เนื่องจากข้อมูลที่ส่งผ่านอินเทอร์เน็ต (และเครือข่ายคอมพิวเตอร์ส่วนใหญ่) จะถูกส่งเป็นก้อนที่จัดการได้ บนอินเทอร์เน็ต ก้อนข้อมูลเหล่านี้เรียกว่า **แพ็กเก็ต (Packet)**

3. แพ็กเก็ตจะผ่าน Application Layer และเดินทางต่อไปยังชั้น TCP แต่ละแพ็กเก็ตจะถูกกำหนด **หมายเลขพอร์ต (Port Number)** เนื่องจากโปรแกรมหลายตัวอาจใช้ TCP/IP Stack และส่งข้อความอยู่ เราจึงต้องรู้ว่าโปรแกรมใดบนคอมพิวเตอร์ปลายทางจะเป็นผู้รับข้อความ เพราะโปรแกรมนั้นจะรอฟังอยู่ที่พอร์ตเฉพาะ

4. หลังจากผ่านชั้น TCP แพ็กเก็ตจะเดินทางไปยังชั้น IP ซึ่งเป็นจุดที่แต่ละแพ็กเก็ตจะได้รับ **ที่อยู่ปลายทาง** คือ 5.6.7.8

5. ตอนนี้แพ็กเก็ตข้อความของเรามีทั้งหมายเลขพอร์ตและ IP Address แล้ว พร้อมที่จะถูกส่งผ่านอินเทอร์เน็ต **ชั้น Hardware** จะทำหน้าที่แปลงแพ็กเก็ตที่มีข้อความตัวอักษรให้เป็นสัญญาณอิเล็กทรอนิกส์และส่งผ่านสายโทรศัพท์

6. อีกด้านหนึ่งของสายโทรศัพท์ **ISP** ของคุณมีการเชื่อมต่อโดยตรงกับอินเทอร์เน็ต เราเตอร์ของ ISP จะตรวจสอบที่อยู่ปลายทางในแต่ละแพ็กเก็ตและกำหนดว่าจะส่งไปที่ไหน โดยมาก แพ็กเก็ตจะถูกส่งต่อไปยังเราเตอร์ตัวถัดไป

7. ในที่สุด แพ็กเก็ตก็ไปถึงคอมพิวเตอร์ 5.6.7.8 ที่นี่ แพ็กเก็ตจะเริ่มจากด้านล่างของ TCP/IP Stack ของคอมพิวเตอร์ปลายทางและเดินทางขึ้นด้านบน

8. เมื่อแพ็กเก็ตเดินทางขึ้นผ่าน Stack ข้อมูลการจัดเส้นทางทั้งหมดที่ Stack ของคอมพิวเตอร์ผู้ส่งเพิ่มเข้ามา (เช่น IP Address และหมายเลขพอร์ต) จะถูกลอกออกจากแพ็กเก็ต

9. เมื่อข้อมูลถึงด้านบนของ Stack แพ็กเก็ตจะถูกประกอบกลับเป็นรูปแบบเดิม คือ "Hello computer 5.6.7.8!"

## โครงสร้างพื้นฐานของเครือข่าย

ISP จะมี **Pool ของโมเด็ม** สำหรับลูกค้าที่โทรเข้ามา ซึ่งถูกจัดการโดยคอมพิวเตอร์บางรูปแบบ (โดยปกติเป็นเครื่องเฉพาะ) ที่ควบคุมการไหลของข้อมูลจาก Modem Pool ไปยังเราเตอร์สาย Backbone หรือสายเฉพาะ การตั้งค่านี้อาจเรียกว่า **Port Server** เนื่องจากมัน 'ให้บริการ' การเข้าถึงเครือข่าย ข้อมูลการเรียกเก็บเงินและการใช้งานมักถูกเก็บรวบรวมที่นี่ด้วย

หลังจากแพ็กเก็ตของคุณเดินทางผ่านเครือข่ายโทรศัพท์และอุปกรณ์ภายในของ ISP แพ็กเก็ตจะถูกจัดเส้นทางไปยัง Backbone ของ ISP หรือ Backbone ที่ ISP ซื้อแบนด์วิดท์มา จากตรงนี้ แพ็กเก็ตจะเดินทางผ่านเราเตอร์หลายตัว ผ่าน Backbone หลายเส้น สายเฉพาะ และเครือข่ายอื่นๆ จนกว่าจะพบจุดหมายปลายทาง คือคอมพิวเตอร์ที่มีที่อยู่ 5.6.7.8

> **ลองดู - โปรแกรม Traceroute**
>
> หากคุณใช้ Microsoft Windows หรือ Unix และมีการเชื่อมต่ออินเทอร์เน็ต นี่คือโปรแกรมอินเทอร์เน็ตที่มีประโยชน์อีกตัวหนึ่ง เรียกว่า **traceroute** และมันจะแสดงเส้นทางที่แพ็กเก็ตของคุณใช้เดินทางไปยังจุดหมายปลายทางบนอินเทอร์เน็ต
>
> เช่นเดียวกับ ping คุณต้องใช้ traceroute จาก Command Prompt ใน Windows ให้ใช้ `tracert www.yahoo.com` จาก Unix prompt ให้พิมพ์ `traceroute www.yahoo.com` คุณสามารถป้อน IP Address แทน Domain Name ได้เช่นกัน
>
> Traceroute จะพิมพ์รายชื่อเราเตอร์ คอมพิวเตอร์ และเอนทิตีอินเทอร์เน็ตอื่นๆ ทั้งหมดที่แพ็กเก็ตของคุณต้องเดินทางผ่านเพื่อไปถึงจุดหมายปลายทาง

## โครงสร้างพื้นฐานของอินเทอร์เน็ต

Backbone ของอินเทอร์เน็ตประกอบด้วยเครือข่ายขนาดใหญ่จำนวนมากที่เชื่อมต่อถึงกัน เครือข่ายขนาดใหญ่เหล่านี้เรียกว่า **Network Service Provider (NSP)** ตัวอย่างของ NSP ได้แก่ UUNet, CerfNet, IBM, BBN Planet, SprintNet, PSINet และอื่นๆ เครือข่ายเหล่านี้เชื่อมต่อกันเพื่อแลกเปลี่ยนทราฟฟิกแพ็กเก็ต แต่ละ NSP จำเป็นต้องเชื่อมต่อกับ **Network Access Point (NAP)** อย่างน้อยสามจุด

ที่ NAP ทราฟฟิกแพ็กเก็ตอาจกระโดดจาก Backbone ของ NSP หนึ่งไปยังอีก Backbone หนึ่ง NSP ยังเชื่อมต่อภายในกันที่ **Metropolitan Area Exchange (MAE)** ด้วย MAE ทำหน้าที่เหมือน NAP แต่เป็นของเอกชน NAP เป็นจุดเชื่อมต่ออินเทอร์เน็ตดั้งเดิม ทั้ง NAP และ MAE เรียกว่า **Internet Exchange Point (IX)** NSP ยังขายแบนด์วิดท์ให้กับเครือข่ายขนาดเล็กกว่า เช่น ISP และผู้ให้บริการแบนด์วิดท์ขนาดเล็กกว่า

## ลำดับชั้นการจัดเส้นทางของอินเทอร์เน็ต

เราเตอร์คือ **Packet Switch** เราเตอร์มักเชื่อมต่อระหว่างเครือข่ายเพื่อจัดเส้นทางแพ็กเก็ตระหว่างเครือข่ายเหล่านั้น แต่ละเราเตอร์รู้จักเครือข่ายย่อยของมันและ IP Address ที่ใช้ เราเตอร์มักจะไม่รู้ว่ามี IP Address อะไรอยู่ 'เหนือ' มัน

กล่องสีดำที่เชื่อมต่อ Backbone คือเราเตอร์ Backbone ของ NSP ขนาดใหญ่ที่อยู่ด้านบนเชื่อมต่อกันที่ NAP ใต้พวกมันคือเครือข่ายย่อยหลายเครือข่าย และใต้เครือข่ายเหล่านั้นก็มีเครือข่ายย่อยอีก ที่ด้านล่างคือ Local Area Network (LAN) สองเครือข่ายที่มีคอมพิวเตอร์เชื่อมต่ออยู่

เมื่อแพ็กเก็ตมาถึงเราเตอร์ เราเตอร์จะตรวจสอบ IP Address ที่วางไว้โดยชั้น IP Protocol บนคอมพิวเตอร์ต้นทาง เราเตอร์จะตรวจสอบ Routing Table ถ้าพบเครือข่ายที่มี IP Address นั้น แพ็กเก็ตจะถูกส่งไปยังเครือข่ายนั้น ถ้าไม่พบ เราเตอร์จะส่งแพ็กเก็ตไปตาม Default Route ซึ่งมักจะขึ้นไปยัง Backbone ถัดไปในลำดับชั้น NSP จนถึงจุดที่เราเตอร์รู้จักว่าจะส่งแพ็กเก็ตไปที่ไหน

## Domain Name และการแปลงที่อยู่

ถ้าอินเทอร์เน็ตต้องอาศัยเพียง IP Address คงไม่ได้รับความนิยมเท่าที่เป็นอยู่ เพราะไม่มีใครอยากจำตัวเลขยาวๆ ดังนั้นจึงมีสิ่งที่เรียกว่า **Domain Name** ซึ่งเป็นชื่อที่มนุษย์อ่านได้และสอดคล้องกับ IP Address

คอมพิวเตอร์จำนวนมากที่เชื่อมต่อกับอินเทอร์เน็ตเป็นโฮสต์ของฐานข้อมูล DNS บางส่วนและซอฟต์แวร์ที่อนุญาตให้ผู้อื่นเข้าถึงได้ คอมพิวเตอร์เหล่านี้เรียกว่า **DNS Server** ไม่มี DNS Server ตัวใดที่มีฐานข้อมูลทั้งหมด พวกมันมีแค่ส่วนหนึ่ง ถ้า DNS Server ไม่มี Domain Name ที่ร้องขอ DNS Server จะส่งต่อคอมพิวเตอร์ผู้ร้องขอไปยัง DNS Server อื่น

เมื่อการเชื่อมต่ออินเทอร์เน็ตถูกตั้งค่า (เช่น สำหรับ LAN หรือ Dial-Up Networking ใน Windows) จะมีการระบุ DNS Server หลัก (Primary) หนึ่งตัวและ DNS Server สำรอง (Secondary) หนึ่งตัวหรือมากกว่า ดังนั้น แอปพลิเคชันอินเทอร์เน็ตใดๆ ที่ต้องการการแปลง Domain Name จะสามารถทำงานได้อย่างถูกต้อง ตัวอย่างเช่น เมื่อคุณป้อน Web Address ลงในเว็บเบราว์เซอร์ เบราว์เซอร์จะเชื่อมต่อไปยัง DNS Server หลักของคุณก่อน หลังจากได้รับ IP Address สำหรับ Domain Name ที่คุณป้อน เบราว์เซอร์จะเชื่อมต่อไปยังคอมพิวเตอร์เป้าหมายและขอหน้าเว็บที่คุณต้องการ

## ทบทวนโปรโตคอลอินเทอร์เน็ต

มีโปรโตคอลทั่วไปจำนวนมากที่ใช้บนอินเทอร์เน็ต เพื่อสรุป:

| ชั้นโปรโตคอล (Protocol Layer)   | โปรโตคอลทั่วไป                      |
| ------------------------------- | ------------------------------------ |
| Application Layer               | HTTP, SMTP, FTP, Telnet              |
| Transmission Control Layer      | TCP, UDP                             |
| Internet Layer                  | IP                                   |
| Hardware / Network Access Layer | Ethernet, Token Ring, Frame Relay    |

## โปรโตคอลชั้นแอปพลิเคชัน: HTTP และ World Wide Web

HTTP เป็น **โปรโตคอลแบบไม่รักษาการเชื่อมต่อ (Connectionless)** ที่ใช้ข้อความ ไคลเอนต์ (เว็บเบราว์เซอร์) ส่งคำขอไปยังเว็บเซิร์ฟเวอร์เพื่อขอองค์ประกอบเว็บ เช่น หน้าเว็บและรูปภาพ หลังจากคำขอถูกให้บริการโดยเซิร์ฟเวอร์แล้ว การเชื่อมต่อระหว่างไคลเอนต์และเซิร์ฟเวอร์ผ่านอินเทอร์เน็ตจะถูกตัดออก จะต้องสร้างการเชื่อมต่อใหม่สำหรับแต่ละคำขอ โปรโตคอลส่วนใหญ่เป็นแบบรักษาการเชื่อมต่อ (Connection-oriented) แต่ HTTP ไม่เป็นเช่นนั้น

เมื่อคุณพิมพ์ URL ลงในเว็บเบราว์เซอร์ จะเกิดสิ่งต่อไปนี้:

1. ถ้า URL มี Domain Name เบราว์เซอร์จะเชื่อมต่อไปยัง DNS Server ก่อนและดึง IP Address ที่สอดคล้องกับเว็บเซิร์ฟเวอร์นั้น

2. เว็บเบราว์เซอร์เชื่อมต่อไปยังเว็บเซิร์ฟเวอร์และส่ง **HTTP Request** (ผ่าน Protocol Stack) เพื่อขอหน้าเว็บที่ต้องการ

3. เว็บเซิร์ฟเวอร์รับคำขอและตรวจสอบหน้าเว็บที่ต้องการ ถ้าหน้าเว็บมีอยู่ เว็บเซิร์ฟเวอร์จะส่งมันกลับ ถ้าเซิร์ฟเวอร์ไม่พบหน้าที่ร้องขอ มันจะส่งข้อความข้อผิดพลาด **HTTP 404** (404 หมายถึง 'Page Not Found')

4. เว็บเบราว์เซอร์ได้รับหน้าเว็บกลับมาและการเชื่อมต่อถูกปิด

5. จากนั้นเบราว์เซอร์จะวิเคราะห์หน้าเว็บและมองหาองค์ประกอบอื่นๆ ที่จำเป็นเพื่อแสดงหน้าเว็บให้สมบูรณ์ ซึ่งมักรวมถึงรูปภาพ แอปเพล็ต เป็นต้น

6. สำหรับแต่ละองค์ประกอบที่ต้องการ เบราว์เซอร์จะสร้างการเชื่อมต่อเพิ่มเติมและส่ง HTTP Request ไปยังเซิร์ฟเวอร์สำหรับแต่ละองค์ประกอบ

7. เมื่อเบราว์เซอร์โหลดรูปภาพ แอปเพล็ต ฯลฯ ทั้งหมดเสร็จแล้ว หน้าเว็บจะถูกโหลดอย่างสมบูรณ์ในหน้าต่างเบราว์เซอร์

## โปรโตคอลชั้นแอปพลิเคชัน: SMTP และอีเมล

**SMTP (Simple Mail Transfer Protocol)** เป็นโปรโตคอลแบบข้อความที่ใช้ในการส่งอีเมล

เมื่อคุณเปิดโปรแกรมอ่านอีเมล สิ่งที่เกิดขึ้นโดยทั่วไปคือ:

1. โปรแกรมอีเมล (Netscape Mail, Lotus Notes, Microsoft Outlook ฯลฯ) จะเปิดการเชื่อมต่อไปยัง Mail Server เริ่มต้น IP Address หรือ Domain Name ของ Mail Server มักถูกตั้งค่าเมื่อติดตั้งโปรแกรมอีเมล

2. Mail Server จะส่งข้อความแรกเสมอเพื่อระบุตัวตน

3. ไคลเอนต์จะส่งคำสั่ง SMTP HELO ซึ่งเซิร์ฟเวอร์จะตอบกลับด้วยข้อความ 250 OK

4. ขึ้นอยู่กับว่าไคลเอนต์กำลังตรวจสอบอีเมล ส่งอีเมล ฯลฯ คำสั่ง SMTP ที่เหมาะสมจะถูกส่งไปยังเซิร์ฟเวอร์ ซึ่งจะตอบสนองตามนั้น

5. การทำธุรกรรมแบบคำขอ/ตอบสนองนี้จะดำเนินต่อไปจนกว่าไคลเอนต์จะส่งคำสั่ง SMTP QUIT เซิร์ฟเวอร์จะบอกลาและการเชื่อมต่อจะถูกปิด

ตัวอย่าง SMTP แบบง่าย:

```
R: 220 BBN-UNIX.ARPA Simple Mail Transfer Service Ready
S: HELO USC-ISIF.ARPA
R: 250 BBN-UNIX.ARPA
S: MAIL FROM:<Smith@USC-ISIF.ARPA>
R: 250 OK
S: RCPT TO:<Jones@BBN-UNIX.ARPA>
R: 250 OK
S: RCPT TO:<Green@BBN-UNIX.ARPA>
R: 550 No such user here
S: RCPT TO:<Brown@BBN-UNIX.ARPA>
R: 250 OK
S: DATA
R: 354 Start mail input; end with <CRLF>.<CRLF>
S: Blah blah blah...
S: ...etc. etc. etc.
S: .
R: 250 OK
S: QUIT
R: 221 BBN-UNIX.ARPA Service closing transmission channel
```

## Transmission Control Protocol

TCP ทำงานดังนี้:

- เมื่อชั้น TCP ได้รับข้อมูลจากโปรโตคอลชั้น Application ด้านบน มันจะแบ่งข้อมูลออกเป็น 'ก้อน' ที่จัดการได้ แล้วเพิ่ม **TCP Header** ที่มีข้อมูล TCP เฉพาะให้แต่ละ 'ก้อน' ข้อมูลที่อยู่ใน TCP Header รวมถึงหมายเลขพอร์ตของแอปพลิเคชันที่ข้อมูลต้องถูกส่งไป

- เมื่อชั้น TCP ได้รับแพ็กเก็ตจากชั้น IP ด้านล่าง ชั้น TCP จะลอกข้อมูล TCP Header ออกจากแพ็กเก็ต ทำการสร้างข้อมูลใหม่หากจำเป็น แล้วส่งข้อมูลไปยังแอปพลิเคชันที่ถูกต้องโดยใช้หมายเลขพอร์ตจาก TCP Header

TCP ไม่ใช่โปรโตคอลแบบข้อความ TCP เป็น **บริการสตรีมไบต์ที่เน้นการเชื่อมต่อและเชื่อถือได้ (Connection-oriented, Reliable, Byte Stream Service)** การเน้นการเชื่อมต่อหมายความว่าแอปพลิเคชันสองตัวที่ใช้ TCP ต้องสร้างการเชื่อมต่อก่อนจึงจะแลกเปลี่ยนข้อมูลได้ TCP เชื่อถือได้เพราะสำหรับแต่ละแพ็กเก็ตที่ได้รับ จะมีการส่ง **Acknowledgement** กลับไปยังผู้ส่งเพื่อยืนยันการส่งมอบ TCP ยังมี **Checksum** ใน Header สำหรับการตรวจสอบข้อผิดพลาด

> **หมายเลขพอร์ตอินเทอร์เน็ตที่ใช้กันทั่วไป:**
>
> | บริการ          | หมายเลขพอร์ต |
> | --------------- | ------------ |
> | FTP             | 20/21        |
> | Telnet          | 23           |
> | SMTP            | 25           |
> | HTTP            | 80           |
> | Quake III Arena | 27960        |

## Internet Protocol

IP ทำหน้าที่จัดการ **การจัดเส้นทาง (Routing)** ของแพ็กเก็ตข้อมูล IP ไม่สนใจเนื้อหาของแพ็กเก็ต มันแค่มองหาขนาดของแพ็กเก็ตและ IP Address เพื่อส่งไปยังจุดหมายปลายทาง

เมื่อชั้น IP ได้รับแพ็กเก็ตจากชั้น TCP ด้านบน มันจะห่อแพ็กเก็ตด้วย **IP Header** ของตัวเอง ข้อมูลสำคัญใน IP Header คือ IP Address ต้นทางและ IP Address ปลายทาง

เมื่อชั้น IP ได้รับแพ็กเก็ตจากด้านล่าง (จาก Network Interface) มันจะตรวจสอบ IP Address ปลายทาง ถ้า IP Address ปลายทางตรงกับ IP Address ของเครื่องนี้ แพ็กเก็ตจะถูกส่งขึ้นไปยังชั้น TCP ถ้าไม่ตรง แพ็กเก็ตจะถูกส่งต่อ (Forward) ไปตามเส้นทาง

## สรุป

ตอนนี้คุณรู้แล้วว่าอินเทอร์เน็ตทำงานอย่างไร แต่มันจะคงอยู่แบบนี้อีกนานแค่ไหน? เวอร์ชันของ IP ที่ใช้อยู่ในปัจจุบัน (เวอร์ชัน 4) อนุญาตให้มีที่อยู่ได้เพียง 2³² ที่อยู่ ในที่สุดจะไม่มี IP Address ว่างเหลืออยู่ **IP เวอร์ชัน 6** กำลังถูกทดสอบอยู่บน Research Backbone โดยกลุ่มพันธมิตรของสถาบันวิจัยและบริษัทต่างๆ

อินเทอร์เน็ตมาไกลมากตั้งแต่จุดเริ่มต้นในฐานะโครงการวิจัยของกระทรวงกลาโหม ไม่มีใครรู้จริงๆ ว่าอินเทอร์เน็ตจะกลายเป็นอะไร แต่สิ่งหนึ่งที่แน่นอนคือ อินเทอร์เน็ตจะรวมโลกเข้าด้วยกันอย่างที่ไม่มีกลไกอื่นใดเคยทำมาก่อน ยุคสารสนเทศกำลังดำเนินไปอย่างเต็มที่
